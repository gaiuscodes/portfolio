How I Built Mtaani Gas

Next.js • PostgreSQL • AWS • Case Study

Mtaani Gas is a digital marketplace connecting households with nearby LPG suppliers. Building it required solving real-world problems — location-based discovery, real-time order tracking, and managing two distinct user types on a single platform.

The Problem

Most households in Kenya still order gas through phone calls and word-of-mouth. Suppliers struggle with visibility, and customers struggle with availability. We needed a platform that would make discovery instant and ordering reliable.

Architecture Decisions

We chose Next.js for the frontend and API routes to keep the stack unified. PostgreSQL with Prisma gave us type-safe database access and easy relationship management between customers, suppliers, and orders. Redis handled session caching and real-time inventory updates.

Location-Based Discovery

The core feature was finding nearby suppliers. We integrated Google Maps API for geocoding and distance calculations. The challenge was performance — calculating distances for hundreds of suppliers on every search would be slow. We solved this with geospatial indexing in PostgreSQL using PostGIS, reducing lookup times from seconds to milliseconds.

Real-Time Order Tracking

Customers expect to see where their order is. We used WebSockets for live status updates — when a supplier accepts or dispatches an order, the customer sees it immediately. This required careful state management and handling connection drops gracefully.

Deployment on AWS

We containerized the application with Docker and deployed on AWS using ECS. PostgreSQL ran on RDS with automated backups, Redis on ElastiCache, and static assets on S3 with CloudFront CDN. The infrastructure-as-code approach with Terraform made environments reproducible.

Lessons Learned

Location-based features require more than just a map — they need careful data modeling, performance optimization, and edge case handling (what happens when a supplier's location isn't precise?). Mobile responsiveness was critical — most users access the platform from phones, not desktops. And finally, building for two user types (customers and suppliers) meant twice the UI complexity but also twice the value.

Mtaani Gas taught me that the best software solves problems people actually have, in ways they can actually use.
